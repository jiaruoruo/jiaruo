## 一、为什么聊这个

## 1.1 行业背景与话题演变

以前聊到最热的话题是EE架构区域化，现在似乎热度下降了，仿佛域到区域的转变已经完成，不得不说有投入确实有产出。因此，新的话题之一自然转向了进一步变革EE架构。

无论话题如何演变，软件定义汽车（SDV）、EE架构优化，还是单纯寻找当前架构下的创新点，大致方向始终是一致的：如何能便于添加新功能、增配新选项，又快又便宜。最终目标，或许是让汽车像手机一样——硬件平台升级后趋于标准化，新功能以软件形式安装、升级，持续创造价值。

仔细想想，当前用户能直接接触到的功能，比如导航、语音助手、音乐视频，其实已经很"SDV"了。装App、OTA升级，这些体验和手机差不多。但车子还有一些不那么显眼但同样重要的部分——车门怎么开、座椅怎么调、灯光怎么控制——这些功能的实现方式，依然保留着比较传统的分布式架构。

## 1.2 SOA服务化的现状与局限

这些年实现的SOA服务化，确实是一项了不起的进步。通过标准化的服务接口，上层应用和底层实现分离开了，服务可以复用，不同团队可以并行开发。但现在的问题是，服务依然是分散部署的。EE架构从分布式演进到域再到区域，但可用的算力依然是分布式的：中央控制器上有一些服务，ZCU上有一些，边缘ECU上可能还有一些。

这种分布式SOA虽然比完全分散要好上不少，但依然带来不少麻烦。从流程角度看，开发要涉及多处，更新服务可能涉及好几个节点，测试要测全链路，出了问题要排查多个地方。从成本角度看，多处的CPU-RAM-Storage组合，就意味着多处冗余的操作系统、中间件和软件。

## 1.3 进一步集中化的需求

那么下一步还能做什么？Zonal架构是否就是终点？

显然不是。例如One-box、One-board到One-chip的演进，体现的也是中央部分的几颗芯片如何进一步集中，如何优化智驾芯片、座舱芯片分散的现状。

那么Zonal和更多的端侧呢？三到四颗Zonal，更多的端侧ECU，多种总线（CAN、LIN）从各个节点连接，然后接各种控制信号。这显然也是过渡形态。

如何将这部分进一步简化和优化？怎么让边缘节点更简单？思路之一也是集中化——不让它们跑控制逻辑，让软件更集中在中央跑，将分散的算力集中到Zonal，或者到算力爆发的中央去。

这催生了MCU-less理念：边缘节点仅保留简单的I/O驱动功能，复杂的控制逻辑统一上移到中央计算平台。

控制逻辑和I/O分离后，它们如何交互？原本的多种总线，以及总线需要的转换网关软硬件，又该如何解决？这又催生了10BASE-T1S和远程控制协议。10BASE-T1S统一了从中央到端的总线，在统一总线基础上，远程控制协议可以进一步简单化，因此催生了RCP协议。

于是，我们的话题就聊到了MCU-less、10BASE-T1S以及RCP。

## 1.4 架构演进的驱动力：双重解耦

集中化并不是目的。目的是双重解耦。

为实现真正的SDV愿景，车载架构需要进一步演进，核心驱动力正是双重解耦。

软硬解耦：将"计算决策"与"执行控制"分离。计算决策运行在高性能CPU上，执行控制通过标准化I/O接口与硬件交互。新增功能不再是"新增硬件"，而是"在现有硬件上新增软件"。

软软解耦：将软件部署在一起，不仅要糅合，更要隔离。避免功能间相互影响、避免升级相互影响、隔离故障传播。这需要线程、进程、虚拟机等技术手段，这又涉及另一个大话题了，SDV不只是把软件放一块。

总之双重解耦的实现依赖于算力集中化。不管谁是因，谁是果，受益于半导体技术的发展，单颗车规级SoC的算力已实现数量级提升，足以支撑原本分散在数十个ECU中的计算任务。

## 1.5 聊回到MCULess, 10Base-T1s和RCP

## MCU-less、10BASE-T1S与RCP的定位

MCU-less：边缘节点仅保留简单I/O驱动，复杂逻辑上移到中央。

统一通信架构：用10BASE-T1S等以太网技术统一替代CAN/LIN/FlexRay等异构总线。

RCP协议：将协议转换逻辑从边缘节点上移到ZCU/中央控制器，边缘节点只需解析简单控制指令。

这些技术不是为了取代SOA，而是为SOA提供更好的通信基础。边缘节点轻量化了，SOA服务自然更容易集中部署。把车载通信做得更简单、更高效，给SDV打个好基础。

## 二、简单聊一下技术

## 2.1 MCU-less

MCU-less的理念很简单：让边缘节点尽可能简单。传统边缘ECU包含完整的MCU，运行操作系统和复杂协议栈；MCU-less架构下，边缘节点只保留最基本的I/O驱动功能，复杂的控制逻辑全部上移到中央计算平台。

这并不是说要完全去掉所有智能，而是把智能集中在资源丰富的地方。边缘节点更像是一个"执行器接口"或"传感器接口"，只负责最基本的信号采集和驱动动作。

边缘节点从含义转换的"小网关"，变成了执行命令的士兵，而且是确定性极强的硬件兵。

## 2.2 10BASE-T1S

## 2.2.1 标准与规范

10BASE-T1S的标准是IEEE 802.3cg，于2020年2月发布。该标准定义了10Mbps单对以太网，支持多点拓扑——这是其能够替代CAN等传统总线的关键特性。从应用角度分析一下标准。

## 2.2.2 PHY架构与子层

以太网PHY对应OSI模型的物理层，内部又分为多个子层：

|      子层       |               功能               |
|---------------|--------------------------------|
|  PHY Control  | PLCA协调器功能、BEACON生成、TO计数、发送接收控制 |
|  PCS（物理编码子层）  |       8b/10b编码（如有需要）、帧定界       |
| PMA（物理介质连接子层） |         串行化/反串行化、时钟恢复          |
| PMD（物理介质相关子层） |         驱动器/接收器、模拟前端电路         |

关键点：

10BASE-T1S使用标准以太网MAC，

而PHY层进行了特殊设计，

以支持PLCA机制和多点总线拓扑。

## 2.2.3 拓扑与连接方式

10BASE-T1S支持两种连接方式：

多点总线连接（推荐用于边缘场景）：多个边缘节点共享同一对双绞线，通过PLCA机制协调数据传输，无需交换机，布线简单，成本低，适配当前的CAN总线拓扑。

点对点连接（适用于特定场景）：以太网全双工通信，适用于带宽要求较高的场景。

## 2.2.4 说到PLCA，它要解决什么问题

先说传统以太网怎么干活：传统以太网用CSMA/CD（载波侦听多路访问/冲突检测）来协调多个节点抢总线。谁想发数据，先听一下——空了就发，忙了就等着。发了之后还得盯着，看有没有别人同时也在发（冲突）。要是撞上了，大家各自等一随机时间再重试。

问题出在哪：这套机制在负载低的时候挺管用。但车上的场景是，多个节点可能时不时就要发数据，负载一上来，冲突就多了。冲突一多，等的时间就没谱了——可能几十微秒，也可能等更久。这个"没谱"对于车上的安全相关功能来说就很要命了。

车上到底要什么：车子上的功能，刹车、转向、安全气囊这些，没一个容得下"不知道什么时候能响应"这种事。系统必须能打包票——最坏情况下响应时间不超过多少毫秒——才能说这个设计是安全的。CAN总线能在车上用这么多年，就是靠位仲裁机制给了这个确定性保证。

PLCA想干嘛：PLCA全称Physical Layer Collision Avoidance（物理层冲突避免），换一种玩法——不让大家抢了，按排好的顺序一个个来。每个节点在每个周期里都有一个专属的时间窗口（TO），轮到你就发，没轮到你老实待着。这样一来，最长等多久完全可以算出来，确定性就有了。

PLCA这套逻辑是放在PHY层实现的。MAC层那边该干嘛还干嘛，根本不知道底下已经换了一套调度方式。正因为这样，现有的以太网软件生态可以直接用，不用大改。

所以一句话总结，PLCA（Physical Layer Collision Avoidance，物理层冲突避免）通过时分复用的方式确保确定性传输。

## 2.2.5 PLCA核心概念

|        概念        |             说明             |
|------------------|----------------------------|
| 协调器（Coordinator） | Node ID = 0的节点，负责发送BEACON帧 |
|     BEACON帧      |      协调器定期发送，启动新的传输周期      |
|     TO（传输机会）     |       每个节点分配的唯一传输窗口        |

## 2.2.6 PLCA配置参数

|   配置项    |            说明            |
|----------|--------------------------|
| Node ID  |      节点唯一标识符（0-31）       |
|  协调器使能   |      Node ID = 0时使能      |
| BEACON周期 | 100μs - 100ms（根据实时性需求配置） |
|   TO长度   |      通常固定（如128字节时间）      |

关键点：

PHY层感知TO并控制发送时机，

MAC层无需感知TO序号，只需在PHY允许时提供数据，

BEACON由协调器的PHY层生成并发送。

## 2.3 RCP

## 2.3.1 定义与起源

RCP（Remote Control Protocol，远程控制协议）由宝马公司于2023年5月在OPEN联盟TC14会议中首次提出，2024年7月OPEN联盟成立TC18：Remote Control小组专门负责标准化工作。

RCP是一种轻量级的通信协议，核心理念是将边缘节点控制器中MCU的协议转换功能迁移至区域控制器ZCU甚至中央控制器中实现。

## 2.3.2 设计动机：解决什么问题

在传统的车载网络架构中，当区域控制器需要控制连接在边缘节点上的执行器时，需要经过一系列复杂的数据转换流程。

传统架构的困境：ZCU节点负责从骨干网接收中央节点的信息（如SOME/IP），转为执行器能够理解的底层接口协议（如SPI、I2C、UART等），或者转换为边缘总线协议（如CAN、LIN），传输到边缘节点。边缘节点的MCU负责解析来自区域控制器的上层协议（如CAN、LIN），并将控制信号转换为执行器能够理解的底层接口协议（如SPI、I2C、UART等）。

这种设计的局限性：一旦更换了执行器的类型或接口，对应节点的软件就需要进行相应的更新和修改。例如，如果更换了一个具有不同SPI映射关系的执行器（原本需要在寄存器0x01中写入0x0A，新执行器需要在寄存器0x0A中写入0x0B），边缘节点就需要修改软件来适配。这种依赖关系增加了系统的维护成本和复杂度。

RCP想解决的问题：把协议转换功能从边缘节点上移到ZCU或中央控制器，让边缘节点只需解析标准化的报文并直接执行控制操作，从而实现硬件无关性和软件可移植性。

## 2.3.3 协议层级定位

RCP是应用层协议，其报文可以根据使用场景封装进不同的传输协议中，热门候选着包括：

|          封装选项           |            说明            |
|-------------------------|--------------------------|
|         SOME/IP         |  将RCP报文封装在SOME/IP的有效载荷中  |
|        IEEE 1722        | 第二层封装，Ethertype = 0x22F0 |
| Google Protocol Buffers |          序列化协议           |

但从问题出发，解决端节点侧效率问题，协议越简化越好，因此1722是更被看好的。

标准化进程：2024年4月，OPEN联盟发布了RCP协议的Draft 0.2版本。2024年7月OPEN联盟正式成立了TC18：Remote Control小组，专门负责RCP协议的进一步开发和标准化工作。目前，关于RCP协议字段的详细规范正在工作组内持续讨论和完善中。

## 2.3.4 工作原理

为了更好地理解RCP协议，我们通过一个具体的控制场景来说明。假设一个执行器通过标准SPI接口连接在边缘节点的MCU上，需要控制这个执行器在其寄存器0x01中写入0x0A。

传统方式（以SOME/IP为例）：区域控制器将SOME/IP报文发送至边缘节点，边缘节点中的MCU解析报文payload中控制执行器的信号，并通过SPI接口修改执行器的0x01寄存器。这一过程中，边缘节点承担了协议转换的重要角色。

RCP方式：区域控制器应用层下发的数据首先在RCP发起端转换为SPI格式的指令，并将这些指令填入RCP协议的payload中。当RCP报文传输至边缘节点后，边缘节点中的RCP模块直接解析payload中的指令，无需进行任何额外的转换工作，即可通过SPI接口直接控制执行器。这种设计将协议转换的职责完全交给了上层控制器，边缘节点只需按照RCP协议标准解析并执行指令。

总线型组播支持：RCP协议的设计还充分考虑了10BASE-T1S总线型连接的特性。由于10BASE-T1S支持总线型拓扑结构，RCP协议允许区域控制器在一条RCP报文中添加多个控制命令，通过组播的方式实现一条RCP报文同时控制总线上多个边缘节点的操作。这种组播机制大大提高了网络通信的效率，减少了总线上的报文数量，降低了通信延迟。实现上不只是组播/广播，需要添加扩展命令，添加节点信息+操作信息。

## 2.3.5 核心价值

轻量化设计：RCP协议最显著的技术特点之一是轻量化设计。与传统的SOME/IP等车载以太网协议相比，RCP协议省去了复杂的会话管理、服务发现和序列化等机制，专注于实现直接的远程控制功能。这种设计使得RCP协议具有更低的协议开销和更快的处理速度，特别适合于对实时性要求较高的控制类应用。同时，轻量化的协议设计也降低了边缘节点的软件复杂度，使得边缘节点能够采用资源受限的硬件平台，从而降低了整体系统成本。

硬件无关性：RCP协议通过将协议转换功能集中在上层控制器中实现，有效解决了传统架构中边缘节点与特定执行器之间的强耦合问题。在采用RCP协议的系统中，当需要更换执行器类型或接口时，只需更新区域控制器或中央控制器中的软件配置，而无需修改边缘节点的任何软件代码。这种硬件无关性大大提高了系统的灵活性和可维护性，降低了产品升级和适配的难度。

软件集中化：RCP协议推动了车载网络中软件集中化的实现进程。在传统架构中，协议转换和信号处理等逻辑分散在各个边缘节点中，导致软件版本管理、功能更新和安全维护等工作面临巨大挑战。通过采用RCP协议，这些功能逻辑被集中在区域控制器或中央控制器中统一管理，实现了软件的集中部署和统一更新。这种集中化不仅简化了软件开发流程，还提高了软件的安全性和可追溯性，为实现软件定义汽车奠定了技术基础。

## 2.3.6 与其他协议的比较

与SOME/IP的比较：SOME/IP是当前车载以太网中广泛采用的面向服务的通信协议，提供了完整的服务发现、消息序列化和远程过程调用等功能。但SOME/IP的设计目标是通用的服务通信，在简单的控制类应用中显得过于复杂。RCP与SOME/IP的主要区别在于：SOME/IP需要边缘节点进行协议转换，而RCP将转换功能上移到区域控制器中；SOME/IP具有完整的会话管理机制，而RCP采用更轻量级的设计；SOME/IP适用于复杂的服务交互，而RCP专注于直接的远程控制。在实际应用中，RCP和SOME/IP并非相互替代关系，而是可以协同工作：在需要复杂服务交互的场景中继续使用SOME/IP，在简单的控制类场景中使用RCP以获得更高的效率。

与CAN/LIN的比较：CAN和LIN总线是传统车载网络中应用最广泛的通信协议，具有成熟的生态系统和广泛的应用基础。然而，随着车辆功能的日益复杂，CAN和LIN总线在带宽、拓扑灵活性和可扩展性方面存在明显局限。RCP协议基于10BASE-T1S以太网技术，在带宽（可达10Mbps）和拓扑灵活性（支持总线型和星型混合拓扑）方面具有显著优势。需要指出的是，RCP协议并非要完全替代CAN/LIN总线，而是与这些传统总线形成互补关系：在车身控制等对带宽要求不高的场景中，CAN/LIN总线仍将是主要选择；在需要高带宽和复杂控制的场景中，RCP协议配合以太网技术将发挥更大优势。

## 三、部署就那么简单吗

## 3.1 MCU-less的部署有几种可能的组合方式：端节点的MCU-less化、ZCU的MCU-less化，以及ZCU加端节点的MCU-less化。

端节点的MCU-less化：端节点向上通过10BASE-T1S连接到ZCU，向下转化为底层接口协议，自身实现RCP协议解析。应用上移部署在ZCU，由ZCU完成以太网骨干网到10BASE-T1S总线结构的转换。这种方式下，ZCU仍然需要承担协议转换的工作，只是边缘节点简化了。

ZCU的MCU-less化：ZCU向上连接以太网骨干网，向下转为边缘总线以及底层接口协议，自身实现RCP协议解析。应用上移部署在中央节点。这种方式不需要关注10BASE-T1S的部署细节，ZCU只负责转发。

ZCU加端节点的MCU-less化：这是更彻底的一种方式，应用上移部署到中央节点。但被远控的节点不只有ZCU，还可能是端节点，这就需要考虑RCP报文在以太网骨干网传输的情况。

这里有两种选择。

点对点方式：端节点通过10BASE-T1S以点对点的方式直接连接到交换芯片。但这样就失去了10BASE-T1S的多点总线特性，线束数量反而增加了，带宽利用率似乎也不会太高。

点对点方式的特点是每个端节点独占一根10BASE-T1S链路，优点是可以使用全双工通信，没有总线仲裁的开销。缺点是需要更多的PHY芯片和连接器，布线成本较高，且无法利用10BASE-T1S的多点总线优势。

总线方式：端节点通过10BASE-T1S以总线形式连接到交换芯片，这是省事省钱的选择。但新的问题来了——标准以太网传输下来的报文，如何和总线结构互联？

PLCA的node0需要作为协调器。如果是一个控制器节点作为node0，比如Zonal控制器，那由软件来构造RCP报文，简单高效。

如果以中央控制器作为远程控制的发起者，那又如何？中央控制器到ZCU，由ZCU转发，可行，但这条通路有软件参与，效率不是最优的。如果直接通过交换硬件转发，如何呢？

这里就涉及RCP报文的转发了。我们以二层报文构造RCP报文，那么如果需要转发到RCP节点的某个节点，就需要交换机端口作为PLCA的node0。

上行可以通信吗？端节点如果将目的MAC设置为中央节点，就需要交换机端口能识别该报文，并进行转发。这里node0端口设置混杂模式是否可行？

下行可以通信吗？如果上行可行，那么交换机端口就可以学习到该节点的MAC，中央控制器向端节点发送，使用端节点MAC即可。如果是总线广播呢？是否可以发送给该端口MAC，该端口通过转发ize自动转为总线广播报文？

如果以上可行，那么整个路径上就不需要软件参与，路径上都可以是MCU-less。

## 3.2 理想与现实的差距

10BASE-T1S加RCP的组合看似完美，但实际部署中还有不少问题需要面对。

首先是成本问题：如果它非常便宜，那毫无疑问可以推广。但硬件成本上，MAC加PHY的价格在十几块钱，远超CAN控制器加CAN收发器的几毛钱。咨询专业人士，这个成本差距在短期内不会消失。

那是节省线束了吗？同区域总线用量相近，并没有因为换成以太网就大幅节省线束。还需要结合具体场景才好详细讨论。

协议栈简化？以太网协议栈不比CAN更简单。但这里处理RCP并不需要复杂的协议管理，如果基于2层的RCP，只需要处理2层报文即可，因此这里算是打平。

边缘节点简化？从协议处理层面算打平，但是因为MCULess的原因，边缘节点大大简化了。

## 四，总而言之

MCULess是好的方向，结合10Base-T1s和RCP，实现更大范围的架构简化，降低产品生命周期总成本。

但实践上，从小范围试点开始。选一个功能简单、节点少的区域，比如某个门的门窗控制，先试试10BASE-T1S加RCP的组合。汽车行业不是互联网，验证周期长，容错空间小。

考虑和标准以太网结合后的端到端部署，全流程的RCP验证。

标准化，虽然RCP标准还没有完全成型，但标准化总是合理的，私有化的协议再去标准化，路径和投入更大。

保持成本预期。10BASE-T1S加RCP在成本上目前没有优势，主要是看长期价值。如果量产规模上来了、成本下降了，这条路自然越走越宽。

总之，EE架构演进是个长期过程，10BASE-T1S和RCP只是其中一个环节。保持关注，保持尝试，但也要有耐心。技术落地从来不是一蹴而就的。
