# 和 AI 协作的一些实践与思考

## 一、开场：触动是怎么来的

今年三月初，我刷到了一篇文章，是 OpenAI 工程团队写的（[https://openai.com/zh-Hans-CN/index/harness-engineering/）。](https://openai.com/zh-Hans-CN/index/harness-engineering/%EF%BC%89%E3%80%82)

文章讲了他们做的一个实验：过去五个月，构建并交付了一款内部产品，其中**没有一行代码是人类写的**。整个代码库——应用逻辑、测试、CI 配置、文档、内部工具——全部由自家的 Codex 生成。团队只有三个工程师，最终跑出了一百万行代码，每人每天平均合并 3.5 个 PR，吞吐量随项目推进还在持续增长。整体交付速度，他们估计是手写代码的十倍。

但数字背后，文章真正在讲的是一套做事方式的转变。

他们发现，AI 在一个"环境不清晰"的项目里会寸步难行——缺工具、缺结构、缺上下文，根本没法推进。工程师的工作重心因此从写代码变成了**给 AI 搭舞台**：整理文档、定义约定、设计反馈机制。他们对工程师角色的定义是："核心任务不再是写代码，而是**设计环境、定义意图、构建反馈回路**。"

文档这件事他们讲得很细。他们试过把所有规范堆进一个大文件，结果失败了——上下文太长，AI 要么漏掉关键约束，要么开始乱优化。后来把文档体系设计成一张**地图**而不是一本百科全书：入口文件只有一百行，只负责指路，细节分散在各自的模块文档里。文章里有一句话几乎是整件事的底层逻辑："**从 AI 的视角来看，任何在上下文中无法被访问的信息，就等于不存在。**"放进代码库的才算数，存在 Google Doc 或者同事脑子里的，对 AI 来说不存在。

随着 AI 的代码吞吐量越来越高，新的瓶颈浮出来了——不是 AI 写得够不够快，而是**人的 QA 注意力跟不上**。于是他们开始让 AI 自己驱动浏览器跑验证、读日志、看监控，把原本需要人盯着的环节也交给了 AI。一个 Codex 任务可以在人类睡觉的时候连跑六个小时，自己修复、重跑、验证，早上起来 PR 已经开好了。

看完之后，我心里落下了一颗种子。这篇文章描述的那个世界，可想象的空间实在太大了。

---

## 二、起点：一个真实的工程场景

有了这颗种子之后，会发现到处都是试验田。

第一个让我停下来认真想的，是我们的数据体系驾驶舱。有一段时间，我们的数据出现了问题，信息有误。排查下来，发现根源在上游——上游的接口悄悄变更了，我们没有感知，数据一直在用旧的逻辑跑，直到客户发现不对才反馈过来。

把这条链路画出来是这样的：

```PlainText
上游 API 悄悄变更
        ↓
   数据持续出错（我们不知道）
        ↓
   客户发现数据异常
        ↓
   客户反馈给我们
        ↓
   我们开始排查
        ↓
   定位到是上游 API 变了
        ↓
   适配代码 → 测试 → 上线
```
整条链路非常长，而且是**后知后觉**——问题早就存在了，我们是最后一个知道的。

如果能主动发现，链路会变成这样：

```PlainText
上游 API 发生变更
        ↓
   监控系统立即检测到
        ↓
   通知推送给我们
        ↓
   我们主动响应
        ↓
   适配代码 → 测试 → 上线
```
中间那段——客户发现、客户反馈、我们排查、定位根因——全部消失了。

在没有那颗种子之前，我对这个想法的回答很简单：做一个工具监控一下就好了。但有了那颗种子之后，脑子里冒出来的问题就变多了：这个需求只有我们有吗？有多少团队在依赖上游接口？如果做成一个产品，能不能解决更多人的这个问题？

## 三、探索：从想法到一个真实的产品

工具和产品的区别，从想法层面就开始了。

工具很好理解：写一个定时任务，固定往目标接口发 GET 或 POST 请求，拿到 response，和上一次的做对比，结构变了就是变更。代码层面几十行能跑起来。

但如果是一个产品，需要想的事情就多了。

```Markdown
┌──────────────────┐         ┌──────────────────────┐         ┌──────────────────┐
│    市场洞察      │ ──────▶ │      协作开发        │ ──────▶ │    运营增长      │
│                  │         │                      │         │                  │
│  · 竞品分析      │         │  · 用户管理与权限    │         │  · 基建 Scaling  │
│  · 差异功能定位  │         │  · 监控引擎设计      │         │  · 拉新与转化    │
│  · 定价策略      │         │  · 多渠道通知        │         │  · 留存与裂变    │
└──────────────────┘         └──────────────────────┘         └──────────────────┘
                                         ▲
                                   今天重点讲这里
```
**前面是市场洞察。** 在动手之前需要先看竞品——已经有了哪些工具，它们的核心功能是什么，用户在哪里不满意，差异点在哪里。我看了 Visualping、Distill.io、Hexowatch、Sken.io，发现了一个空白：这些竞品几乎全部聚焦在网页内容监控，没有人认真做 API 的 JSON 结构对比。而这恰好是工程师最高频的痛点。定价上参考市场中位，找到一个既能覆盖成本又有竞争力的区间。

**中间是协作开发，** 这是今天的重点，后面单独展开。简单说，做一个产品涉及的东西包括：用户要能注册登录，权限要控制好谁能看什么；监控要设计得通用，不只是我们这一个接口，任何用户都能配置自己的目标地址和请求方式，频率要可以灵活设定；通知是一个关键模块，不同用户用不同的工具，飞书、邮件、Telegram、Slack、短信，渠道越多覆盖面越广。

**后面是运营增长。** SaaS 产品上线之后，运营层面需要关心的事情大致分两类：一是基础设施的健康，包括可用性、性能、成本，以及用户规模增长后的 Scaling 方案；二是商业增长，销售手段上有几条成熟路径——内容营销和 SEO 做长期流量沉淀，PLG（产品主导增长）靠免费层自然获客再向付费转化，社区运营在开发者群体里建立口碑，以及渠道合作借助集成平台的流量。

今天我们重点讲中间这部分——协作开发，以及在这个过程里我是怎么和 AI 一起工作的。

## 四、工程：AI 协作的实际框架

做这个产品之前，我没有想清楚 AI 协作会遇到什么问题。做完之后，回头看，可以归结为三件事：流程怎么设计、记忆怎么管理、成本怎么控制。

```PlainText
┌─────────────────────────┬──────────────────────────┬─────────────────────┐
│        Workflow         │        Long-term          │        Cost         │
│        流程             │        记忆与一致性        │        成本         │
├─────────────────────────┼──────────────────────────┼─────────────────────┤
│  · 正向工作流           │  · 冷启动协议             │  · 强/弱模型分工    │
│  · Feature Spec         │  · Reusable 文档索引      │                     │
│  · 禁止清单与角色边界   │  · 状态由人维护           │                     │
└─────────────────────────┴──────────────────────────┴─────────────────────┘
```

### 4.1 Workflow：流程怎么设计

现在的 AI 有非常强的规划和执行能力，强到有时候会让你觉得「它可以自己搞定一切」。但在一个具体的产品里，这种能力如果没有约束，会出问题——它会按照自己对需求的理解往前走，理解多了就过度设计，理解少了就做残了；执行上也一样，没有边界它会顺手改很多东西，每次都不一样，最终变得不可预测。

我想要的不是「最强」的 AI，而是「刚刚好」的 AI——对需求的理解不多也不少，对执行的范围不宽也不窄。围绕这个目标，Workflow 这块设计了三件事：正向工作流、Feature Spec、禁止清单。

#### 4.1.1 正向工作流

**是什么**：一套让 AI 进入项目就知道「该从哪里读、该做什么、怎么做」的导航设计，从 README 开始，指向 `AI-GUIDE.md`，再展开到具体的任务规范。

**在哪维护**：`README.md` 作为入口，`docs/AI-GUIDE.md` 作为执行规程。

**具体是这样的**：

AI 读一个项目，会先读 README——这和人一样。所以 README 是两种读者共用的，但我在里面做了区分。README 的主体是一张**文档索引**，把整个项目的文档结构列出来，任何人来了都能找到方向：

```PlainText
产品文档   → docs/product/
页面定义   → docs/pages/
开发规范   → docs/development/features/
可复用资源 → docs/reusable/
测试       → docs/testing/
基础设施   → docs/INFRA.md
```
然后在工作流部分的末尾，专门加了一行：

```PlainText
AI 协作详见 docs/AI-GUIDE.md
```
人看到这行，大概知道「有一份 AI 用的文档」；AI 看到这行，会去读它——这是分叉点。

`AI-GUIDE.md` 里面说了两件事。第一件是工作流，就是前面那条单向流水线，规定了阶段顺序和不能跳步。第二件是**任务类型规范**——把 AI 在这个项目里可能接到的任务分成了四类：开发任务、测试任务、验收任务、代码审查任务，每一类都有固定的执行步骤，比如开发任务：

```PlainText
1. git pull
2. 读 docs/pages/<页面>.md — 了解页面功能和权限
3. 读 docs/development/features/<功能>.md — 了解改动要求
4. 读 docs/reusable/ — 查找可复用组件/接口/工具
5. 写代码
6. git add -A && git commit -m "feat: <功能名>" && git push
7. 说"开发完成，已推送"
```
步骤是固定的，输入从哪里来、产物是什么、做完说什么——全部写死。AI 接到开发任务，就按这个步骤走，不需要自己规划，也不会走偏。

#### 4.1.2 Feature Spec

**是什么**：每个功能对应一个独立的 `.md` 文件，是 AI 编码时唯一的需求来源。需求不从对话里传，从文档里传。

**在哪维护**：`docs/development/features/` — 这个项目里最终写了 23 份 Spec，对应 23 个可独立交付的功能。

**具体看一个例子**——`features/dashboard-redesign.md`：

```PlainText
## Schema 变更
Change 表新增字段：
  acknowledgedAt  DateTime?   // 已确认时间，null = 未确认
  acknowledgedBy  String?     // 确认人 userId

## 后端改动
GET /api/changes
  - 新增 acknowledged 字段过滤支持
  - 返回数据新增 acknowledgedAt 字段

POST /api/changes/[id]/acknowledge  （新接口）
  - 标记单条变化为已确认
  - 需验证该 Change 属于当前用户有权限的 Monitor

## 前端改动
src/app/dashboard/page.tsx 全面重构
  - 新增 RecentChangesFeed 组件（固定 10 条，含已确认 checkbox）
  - 新增 MonitorStatusPanel 组件（最近活跃 10 个 Monitor）
  - 移除 AiGreeting 组件

## 验收标准
- [ ] 3 个指标卡片正确显示（Active / 今日变化 / Error数）
- [ ] ERROR Monitor 独立区块，无 ERROR 时隐藏
- [ ] 单条"已确认"点击后变灰，不消失
...（共 11 条）
```
「正确」的定义不是「功能上说得过去」，而是「所有验收标准都打了勾」。Spec 里写的每一条都是可以明确打勾或打叉的行为断言。

#### 4.1.3 禁止清单与角色边界

**是什么**：对每种任务类型明确列出 AI 不能做的事，防止它「顺手做了不该做的事」。

**在哪维护**：`docs/AI-GUIDE.md` — 每种任务类型下方都有独立的 ❌ 禁止项列表。

**具体内容**，以开发任务为例：

```PlainText
禁止：
❌ 读 testing/ 目录
❌ 自己写测试
❌ 修改 todo.md / done.md（状态由人类维护）
❌ 擅自加 npm 依赖
❌ 改 Prisma schema（除非任务要求）
```
测试任务禁止修改 `features/` 里的业务逻辑来凑测试通过；代码审查任务禁止直接重构代码，只输出报告等待人类决策。背后的逻辑是：AI 会做它被允许做的事，也会做它没有被明确禁止的事。允许清单和禁止清单同等重要。

### 4.2 Long-term：记忆怎么管理

AI 没有持久记忆，每次新会话对这个项目一无所知。不解决这个问题，时间越长，重复造轮子和风格漂移就越严重。解法是把记忆外化成文档放进代码库，分三层：冷启动协议、Reusable 文档索引、状态管理。

#### 4.2.1 冷启动协议

**是什么**：每次新会话，AI 必须先读的两个文件，用来恢复对项目的完整认知。

**在哪维护**：`docs/AI-GUIDE.md` + `docs/CONTEXT.md`。`AI-GUIDE.md` 的开头第一行就是「进入项目第一件事读这个文件，然后读 CONTEXT.md」。

**两个文件各自负责什么**：

`CONTEXT.md` 是产品知识——这个产品是什么、套餐配额的权威数值、数据模型长什么样、核心代码约定。举个具体的约定：

```TypeScript
_// 获取当前登录用户（Server Component / API Route）_import { getServerUser } from "@/lib/server-auth";
const user = await getServerUser();

_// 日志（不用 console.log）_import { createLogger } from "@/lib/logger";
const log = createLogger("module-name");
```
这些约定写在 `CONTEXT.md` 里，AI 每次进来读完就知道该用哪个函数、不该用哪个。

`AI-GUIDE.md` 是行为规程——当前是什么任务类型、该做什么、不该做什么、每类任务的步骤是什么。两个文件加起来不到两百行，覆盖了一个新 AI 进来需要知道的全部上下文。

#### 4.2.2 Reusable 文档索引

**是什么**：记录项目里已有的可复用资产，防止 AI 在不知道某个东西存在的情况下重复造轮子。

**在哪维护**：`docs/reusable/` — 五个文件，分别对应不同类型的资产：

```PlainText
components.md   前端组件（MonitorForm / DiffViewer / ScreenshotDiff / TrendChart 等）
api-routes.md   后端接口（Monitor / Changes / 通知渠道 / 支付 / 组织 等完整路由表）
lib.md          工具函数（auth / logger / notifier / prisma / redis 等）
config.md       环境变量（必填项 / 可选覆盖 / 三套环境对比）
ui-patterns.md  UI 交互规范（Toast 怎么用 / Modal 怎么触发 / Loading 态处理）
```
举个例子，`api-routes.md` 里记录了 Monitor 相关的全部接口，包括 `GET/POST /api/monitors`、`PATCH/DELETE /api/monitors/[id]`、`POST /api/monitors/[id]/check` 等十几条。AI 开发新功能时先查这份索引，发现接口已经存在就直接调，不存在再新建。`AI-GUIDE.md` 里写明：开发任务开始前必须查这份索引，功能完成后必须更新它。

#### 4.2.3 状态由人维护

**是什么**：任务的进度和归档状态只由人来写，AI 只能读，不能改。

**在哪维护**：`docs/development/todo.md`（待办）+ `docs/development/done.md`（归档）。`todo.md` 里每条任务有三种状态：❌ 未开始 / 🔧 开发中 / ✅ 完成。

`**AI-GUIDE.md**`** 里明确写着**：

```PlainText
禁止：
❌ 修改 todo.md / done.md（状态由人类维护）
```
任务完成了、进展到哪一步了、什么时候归档，这些是人的职责。这个设计背后的逻辑是：状态一旦被 AI 写坏了，排查成本很高；让人来维护状态，成本低、可控、出了问题一眼能看出来。AI 执行，人维护状态，职责不混。

### 4.3 Cost：成本怎么控制

#### 4.3.1 强 / 弱模型分工

做一个产品和做一个实验不一样——资源不是无限的。API 调用是按 token 计费的，如果所有任务都用最强的模型，成本会很快失控；但如果全用弱模型，复杂任务的质量撑不住，省了钱但出了问题。

价格差距比你想象的大。以这个项目用的模型为例，Claude Sonnet 4.5 和 MiniMax 2.5 的价格差了大约 10 倍。同样跑 100 个任务，模型选对了，成本可以差出一个数量级。

**是什么**：按任务类型匹配模型——判断类任务用强模型，执行类任务用弱模型。在有限的资源下，把钱花在真正需要判断的地方。

**在哪维护**：`docs/AI-GUIDE.md` — 任务类型章节里注明了哪些任务「强模型执行」。

**具体分工**：

```PlainText
任务类型           模型    原因
──────────────────────────────────────────────────────
写 Feature Spec   强模型  需要理解页面文档拆解成精确编码指令
验收任务           强模型  需要对全局代码做判断，输出影响项目的结论
代码审查           强模型  需要识别重构问题，更新 reusable 文档

开发任务           弱模型  上游有 Spec，任务明确，执行为主
测试任务           弱模型  读测试文档写用例，执行为主
```
为什么 Feature Spec 必须由强模型来写——因为它是编码的唯一依据，一旦写错，弱模型按照错误的 Spec 执行，产出的东西全错了，而且错得很难发现。写 Spec 的成本花在强模型上是值得的；写完之后的执行交给弱模型，量大、重复、不需要全局判断，正是弱模型擅长的地方。

---
两周，一个人，644 个 commit：完整的 SaaS 产品、多通知渠道、团队权限体系、支付订阅、测试覆盖。这不是说 AI 有多厉害，而是说——**一套设计清晰的协作框架，能让 AI 的产出变得可预测、可控、可持续。**

## 五、诚实的总结

回头看这两周，有些东西确实有效，有些东西没有被解决。

有效的部分里，文档作为上下文基础设施是整个框架最核心的一条，没有它其他的设计都是空的。任务颗粒度的控制也很重要，每批 2–3 个功能是我找到的最佳节奏，太多了 AI 容易在任务之间串扰，太少了效率不够高。强弱模型的分工成本控制效果明显，这个不展开了，数字前面说过了。还有一个容易被忽视的是 git 提交作为协作日志——每个 commit 对应一个 Feature Spec，出了问题知道回溯到哪里，问题可以定位。

但也有些事情我没有解决。AI 无法感知上下文窗口以外的信息，这意味着如果多个 AI 在并行工作，它们互相不知道对方在改什么，冲突很难避免。文档维护是一个持续的成本，文档一旦过期，AI 拿到的上下文就是错的，它会基于过期信息做判断，这个问题没有银弹，只能靠纪律。还有验收，AI 写的测试有时候是「自己测自己」——它会按照它理解的需求写测试，而它理解的需求本身可能就有偏差，所以真实验收还是需要人来做。

最后如果用一句话来总结：**AI 无法替代工程师，但能让一个工程师做到小团队能做的事——前提是你愿意花时间设计协作方式，而不只是把 AI 当搜索框用。**

以上是我在这段探索里沉淀出来的一套方法论。它是可以跑起来的，但不够高效，还有很多没解决的问题，也有很多我自己还没想清楚的地方。

如果你在工作里也有类似的想法，或者已经在做类似的尝试，很希望有机会交流。
