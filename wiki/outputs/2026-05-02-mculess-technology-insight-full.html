2.2 看客户

2.2.1 汽车领域客户

目标用户画像

  • 用户类型：主流整车 OEM（吉利/长安/理想/小鹏等）及其 Tier1 供应商（安波福/博世/大陆等）
  • 规模：年产量 10 万辆以上的量产平台；2026–2028 新平台立项阶段的架构决策团队
  • 核心使用场景：车身域控制器（门锁/座椅/氛围灯/天窗/雨刮），ASIL-B 以下节点

核心痛点（Top 3）

  1. 痛点一：整车 OTA 涉及 50–100 个边缘节点固件，维护成本极高，任一节点刷写失败都可能影响交车；固件版本管理混乱导致召回风险持续存在
  2. 痛点二：点对点布线导致线束总长 1500–2000m，线束占整车成本显著；新能源车每减重 1kg 可提升续航约 0.3%，但传统架构减重空间极小
  3. 痛点三：新功能开发周期 12–24 个月，每次功能迭代需硬件改版 + RTOS/固件重开发 + 功能安全重认证，严重制约 SDV（软件定义汽车）战略落地

MCU-less 对 SDV 核心诉求的支撑：

SDV 核心诉求MCU-less 如何支撑
OTA 功能迭代边缘零固件 → OTA 复杂度从"整车百节点"降至"ZCU/CCU 数节点"
功能按需订阅ZCU 侧软件控制 → 同一硬件平台可动态激活不同功能
软硬件解耦RCP 芯片标准化 → 更换 ZCU 软件不影响边缘硬件，反之亦然
快速迭代边缘无软件 → 新功能仅需 ZCU 侧软件升级，无需重新认证边缘固件

功能安全适用边界（客户最关心的红线）：

应用域MCU-less 适用性原因
车身域（门锁/座椅/照明/雨刮）✅ 强烈推荐控制逻辑简单，无本地安全计算需求
智能电源域✅ 推荐电源路由逻辑可由硬件承担
热管理系统✅ 可用多执行器协调，逻辑可上移
底盘气悬域⚠️ 不推荐本地控制算法复杂，域控 CPU +10% 负载
制动/转向/气囊（ASIL-D）❌ 禁止必须本地降级（Fallback），MCU 不可去除

结论：MCU-less 的适用范围是车身舒适域（Body Domain），ASIL-B 以下场景。ASIL-D 功能安全系统必须保留本地 MCU 作为降级保障。

2.2.2 机器人领域客户

目标用户画像

  • 用户类型：人形机器人整机厂（宇树/优必选/小米机器人/特斯拉机器人等）及关节模组供应商
  • 规模：追求 ¥99,999 量产价格目标的整机厂；外设节点密度 >30 个的商用机型（如 UBTECH Walker X 的 200+ 节点）
  • 核心使用场景：皮肤触觉传感器阵列（32–128 路 IO）、LED 状态灯、散热风扇群、夹爪末端执行器（立即可落地）；关节模组电机控制（2027–2028 拐点后）

核心痛点（Top 3）

  1. 痛点一：关节模组每颗 MCU + FOC 固件需独立开发维护，30–50 套固件碎片化，OTA 需逐一刷写，新算法 TTM 达 6–12 个月
  2. 痛点二：关节控制板 PCB 30×40mm，每个手指关节 MCU + 驱动芯片体积超过关节可用空间（<10mm×15mm），是灵巧手小型化的直接障碍
  3. 痛点三：臂部 7 个关节每关节需 8–14 根线缆穿越折弯处，整机线束 1.5–2 kg，每降 100g 可提升末端负载能力约 10%，线束是关键重量瓶颈

外设节点类型当前方案MCU-less 替代方案收益
皮肤/触觉传感器阵列多个分布式 MCU 采集 + CAN 上传RCP 节点直接转发 ADC/GPIO 数据PCB 减小 50%+，重量降低
状态 LED / 显示条MCU + SPI 控制 LED 驱动RCP 节点直接控制 LED Driver零固件，OTA 复杂度归零

电池管理子模块MCU + AFE 芯片ASIC + 10BASE-T1S 上报安全可靠性提升

2.4 看自己

✅ 我们的优势（以国内 Tier1/整机厂视角）

  • 优势 1：本土协同优势——与汇顶 GE1101 等国产芯片处于同一生态圈，联合开发响应速度快于国际 Tier1；联合 POC 验证中（理想汽车/宇树科技等）
  • 优势 2：架构迁移能力——已具备 ZCU SOA 服务化开发经验（参考小鹏 XEEA3.5 的 300+ 原子化服务理念），可直接复用于 MCU-less 控制逻辑上移
  • 优势 3：双赛道覆盖——汽车车身域 + 机器人外设节点底层技术高度共通（100BASE-T1 + RCP + SOA），一套能力体系同时覆盖两个市场，投入产出比极高
  • 优势 4：成本优势——汇顶 GE1101（~14 RMB）相比 ADI AD3304（~$1.3 ≈ 9–10 元）差距已缩小，且 IO 密度（120 路 vs 12 路）优势显著
  • 优势 5：汽车→机器人迁移路径已验证——小米汽车（YU7 线束减少 40%）→ 机器人团队直接复用工程积累；小鹏汽车 XEEA3.5（线束减重 30%）→ 小鹏鹏行机器人沿用 10BASE-T1S 规划

⚠️ 我们的短板

  • 短板 1：RCP 中间件能力薄弱——传统嵌入式固件团队尚未向"服务化软件工程师"完成转型；ZCU SOA 服务开发需要全新能力体系
  • 短板 2：功能安全认证路径不清晰——MCU-less 节点 ASIL-B 认证工具链缺失；认证边界从"逐节点认证"→"ZCU 层整体认证"的迁移工程成本高，将成为 OEM 采购的摩擦点
  • 短板 3：机器人专用验证不足——100BASE-T1 在关节动态弯折/振动环境（每天 10,000–100,000 次）下的可靠性尚无专项数据；AEC-Q100 汽车标准无法直接适用
  • 短板 4：10BASE-T1S 工具链不成熟——与 CAN 相比，调试工具链（示波器/协议分析仪）均不完善，客户导入摩擦大

3.1 技术方案详解

3.1.1 总体架构：双重解耦

MCU-less 的核心价值不仅是"去掉一颗 MCU"，而是实现两个层面的解耦：

[软硬解耦]  硬件标准化 + 功能软件化 → 支持 OTA 升级而无需硬件改版
[软软解耦]  应用逻辑 ⊥ 通信协议栈 → 微服务化部署，通过标准接口调用边缘 IO

技术本质：

  • 边缘节点仅保留 MCU-less 芯片（硬件状态机）+ 驱动芯片（HSD/H-Bridge/LED Driver）
  • MCU-less 芯片内部无操作系统、无应用程序，通过硬件电路直接执行 ZCU 下发的 GPIO/PWM/SPI/I2C 指令
  • ZCU/CCU 承担所有控制逻辑和状态机，形成完全集中式控制

传统 ECU → MCU-less 节点的硬件革命：

传统ECU节点：MCU + 晶振 + 复位电路 + 存储器(Flash/EEPROM) + 电源管理 + 通信收发器 + 驱动芯片（HSD/H-Bridge/LED Driver）
                          ↓ MCU-less
MCU-less 节点：RCP 芯片（含 MAC-PHY + 硬件状态机）+ 驱动芯片（HSD/H-Bridge/LED Driver）

3.3 关键指标

⚡ 性能指标

  • 总线延迟：100BASE-T1 路线 ~50 μs（汇顶 GE1101）；10BASE-T1S 路线 <1 ms（ADI AD3304）
  • 节点 IO 密度：100BASE-T1 路线 120 路/节点（GE1101）；10BASE-T1S 路线 10–20 路/节点
  • 通信速率：100BASE-T1 = 100 Mbps 全双工；10BASE-T1S = 10 Mbps 半双工；CAN FD Light = 500K–2 Mbps
  • 节点数上限：10BASE-T1S 最多 255 节点（典型 8–30）；100BASE-T1 支持 60 节点
  • 机器人关节位置环延迟：10BASE-T1S + RCP 可满足 1–4ms（250–1000 Hz 外环命令）
  • 机器人外设控制延迟：10–100ms，10BASE-T1S / 100BASE-T1 均完全满足

🛡️ 质量指标

  • PCB 面积缩减：汽车单节点 PCB 从 25×35mm → 15×20mm（约 51%）；机器人关节 PCB 从 30×40mm → 15×20mm（约 56%）
  • 固件归零：边缘节点固件数量从 50–100 套 → 0，整体 100% 消除
  • 线束减重目标：整车 30–60%（小米 YU7 实现 40%，小鹏 XEEA3.5 实现 30%）；机器人关节走线减少 70%
